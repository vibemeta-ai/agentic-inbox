# Agentic Inbox Runtime Evidence plan

**Status:** Local protocol verified; no deployed observation exists

**Target:** `inbox.vibe-meta.ai`

**Scenario:** synthetic order request `VM-1007`

This plan makes the first reference-product evidence repeatable without customer data, external delivery or an unbounded failure hook. Every mutation still requires explicit authorization for the exact deployed commit.

## Evidence boundary

The teaching control plane is protected by both Cloudflare Access and `TEACHING_ADMIN_TOKEN`. It is available only when the reference configuration contains exactly one allowlisted `lab@<configured-domain>` mailbox.

| Control | What it does | What it proves | What it does not prove |
| --- | --- | --- | --- |
| `POST /reset` with `normal` | Deletes known synthetic artifacts, resets state and Agent history, then runs the normal inbound function with automatic Agent work enabled | Deployed intake, attachment persistence, Agent dispatch and resulting product state | Internet email delivery, provider delivery SLA or customer outcome |
| `POST /replay` | Re-enters the deployed inbound function with the exact same synthetic MIME payload and suppresses any new Agent scheduling | Application replay identity and convergence in the deployed Worker | A duplicate generated by Cloudflare Email Routing itself |
| `POST /conflict` | Re-enters the deployed inbound function with the same Message-ID and a changed body | Changed-payload conflict recording without replacing the original work | Malicious-email detection or general content security |
| `POST /reset` with `draft_failed` | Runs normal intake without Agent scheduling, then injects one named `draft_failed` transition | Deployed failure projection, operator next action and retry boundary | A natural Workers AI, network or Agent implementation failure |
| `POST .../retry-draft` | Atomically claims the failed operation, persists its trigger and schedules the real deployed Agent through a MailboxDO alarm | Guarded recovery across the API request boundary, competing-request rejection and eventual Draft result | External send, approval or delivery convergence |
| Anonymous read | Requests the deployed product without an Access session | The observed edge/Worker authorization response | Per-mailbox authorization beyond the configured shared Access boundary |

Local tests separately prove that a real non-2xx Agent trigger response becomes `draft_failed`. Keep that claim classified as Teaching Implementation until an actual deployed trigger failure is observed without unsafe production fault injection.

## Prerequisites

Record before any scenario mutation:

- full 40-character teaching commit;
- Cloudflare Worker version and deployment timestamp;
- reference configuration compatibility date and bindings;
- approved operator and evidence window;
- successful anonymous-access check;
- confirmation that `EMAIL_ADDRESSES` contains only `lab@vibe-meta.ai`;
- confirmation that all inputs and stored artifacts are synthetic.

The examples below assume the request already passes the approved Cloudflare Access mechanism. Access credentials, cookies and service-token headers are intentionally omitted and must never appear in captured output.

```bash
export BASE_URL=https://inbox.vibe-meta.ai
export TEACHING_ADMIN_TOKEN='<redacted-local-shell-value>'
```

Do not paste the exported value into a terminal recording, screenshot, lesson body or evidence artifact.

## Capture sequence

### 1. Authorization boundary

From a context with no Access session, record only the status, redirect target class and timestamp:

```bash
curl -sS -o /dev/null -D - "$BASE_URL/"
curl -sS -o /dev/null -D - \
  "$BASE_URL/api/v1/teaching/scenarios/vm-1007/status"
```

Accept the response actually observed. Depending on Access configuration, this may be a redirect or `403`. Do not normalize the result into an expected value after capture.

### 2. Normal intake and Agent work

```bash
curl -sS -X POST \
  "$BASE_URL/api/v1/teaching/scenarios/vm-1007/reset" \
  -H "Authorization: Bearer $TEACHING_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{"mode":"normal"}'

curl -sS \
  "$BASE_URL/api/v1/teaching/scenarios/vm-1007/status" \
  -H "Authorization: Bearer $TEACHING_ADMIN_TOKEN"
```

Record the returned `emailId`, operation ID, external identity, state, intake attempts, Agent attempts, `agentTriggerPending`, current Draft ID and conflict count. A normal reset may first return the durable queued `drafting` state; poll with a bounded operator timeout until it reaches review or an explained failure.

### 3. Exact application replay

```bash
curl -sS -X POST \
  "$BASE_URL/api/v1/teaching/scenarios/vm-1007/replay" \
  -H "Authorization: Bearer $TEACHING_ADMIN_TOKEN"
```

Acceptance requires the same operation ID and email ID, unchanged conflict count, no extra message or attachment projection and no new Agent attempt caused by the replay.

### 4. Changed-payload conflict

```bash
curl -sS -X POST \
  "$BASE_URL/api/v1/teaching/scenarios/vm-1007/conflict" \
  -H "Authorization: Bearer $TEACHING_ADMIN_TOKEN"
```

Acceptance requires the original operation and payload hash to remain authoritative while `conflictCount`, `lastConflictPayloadHash` and `lastConflictAt` record the rejected changed delivery.

### 5. Injected Draft failure

This is another destructive reset and requires its own approval:

```bash
curl -sS -X POST \
  "$BASE_URL/api/v1/teaching/scenarios/vm-1007/reset" \
  -H "Authorization: Bearer $TEACHING_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{"mode":"draft_failed"}'
```

Acceptance requires `state=draft_failed`, `agentAttempts=0` and `lastError=synthetic_teaching_draft_failure`. The UI must show an actionable Retry Draft state. Describe this as an injected recovery scenario, never as a naturally observed model outage.

### 6. Guarded retry and one current Draft

Use the `emailId` returned by the failed reset:

```bash
curl -sS -X POST \
  "$BASE_URL/api/v1/mailboxes/lab%40vibe-meta.ai/emails/<email-id>/operation/retry-draft"
```

The accepted request returns `202` with `state=drafting`, `agentAttempts=1`, `agentTriggerPending=true` and a cleared error. A competing request during the same attempt must return `409`. Poll the Admin status until the durable alarm produces `awaiting_human_review` or a new explained `draft_failed` state.

When it reaches review, verify that the operation has one `currentDraftId` and that the Draft folder contains exactly one current Draft for the synthetic turn:

```bash
curl -sS \
  "$BASE_URL/api/v1/mailboxes/lab%40vibe-meta.ai/emails?folder=draft"
```

This is still a proposal boundary. Do not click Send and do not describe the customer request as completed.

## Redaction and cleanup

- Remove Access identities, JWTs, cookies, service-token headers and the teaching Admin token.
- Remove raw email headers that could contain infrastructure identifiers not required by the claim.
- Keep only `example.test` identities and the approved `lab@vibe-meta.ai` address.
- Capture the smallest useful API projection or UI region; do not export the entire Durable Object database.
- After the approved window, enumerate the attachment manifest and R2 prefix before running the destructive reset.
- Record cleanup as an operational action, not as proof that Durable Object storage was physically erased.

## Stop conditions

Stop the capture and preserve the current state when:

- the deployed commit or Worker version differs from the approved identity;
- more than one teaching mailbox is configured;
- a request contains non-synthetic data;
- exact replay changes the original operation identity or schedules another Agent attempt;
- conflict replaces the original payload;
- retry produces multiple current Drafts;
- the system cannot distinguish `drafting`, `draft_failed` and `awaiting_human_review`;
- Access or the teaching Admin boundary does not fail closed.

Do not repair a failed observation in place and publish only the successful retry. Record the failure, decide whether a new deployment is required and begin a new evidence record after authorization.
